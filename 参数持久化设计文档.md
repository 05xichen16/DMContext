# 配置参数支持动态配置和持久化 - 设计文档（C++ / DMContextService）

> 实施语言：C++（在本服务内实现，不引入新进程）
> 持久化后端：现有 `RagMgr`（沿用 `tbl_` 命名）
> REST API 路径：完全沿用现有 `route_mgr.cpp` 注册的路径，不变
> 适用范围：本方案只搭建"动态配置 + 持久化"的基础设施。具体业务参数（如 `first_query_db_top_k`、`memory_wait_timeout_ms` 等 37 项）的拆分与默认值落地，**不在本方案内**，作为后续工作

---

## 一、需求背景

当前 `DMContextService` 的配置加载与持久化存在三个问题：

1. **持久化方式脆弱**：`ConfigMgr::SaveConfigToFile()` 通过 temp-file-then-rename 重写 `dmcontextservice_config.json`。运维如果替换或升级了配置文件，运行时通过 REST 接口改过的值会被覆盖。
2. **没有版本管理**：当前文件没有 `version` 字段，无法识别"文件升级"和"用户修改"两个场景的差异。
3. **首次启动与重启行为不一致**：用户改过的值只活在文件里，重启读到的就是文件最后写入的状态，缺少"用户修改值"和"文件默认值"的区分。

本方案目标：

1. 业务代码统一从**内存**读配置，O(1)。
2. REST 写入时**先更新内存、异步落 DB**。
3. 配置文件 `version` 升级时执行 reload：**用户改过的项保留，未改过的项跟随文件升级，新增的项插入**。
4. REST API 路径保持现状，外部调用方无感知。

## 三、方案概述

**一句话**：配置文件加 `version` 字段；启动时按 version 装载到内存，并同步到 RagMgr DB；运行时业务统一从内存读 `override_value`，REST 写入时先更新内存再异步落 DB；文件 `version` 升级时按"未改的同步、改过的保留"的算法执行 reload。

**两类配置**：

| 类型 | 文件 | DB 表 | id 形态 |
|---|---|---|---|
| common | `dmcontextservice_config.json` | `tbl_dm_context_common_config` | `{type}:{name}`，例 `query_config:mode` |
| prompt | `prompt.json` | `tbl_dm_context_prompt_config` | `prompt:{name}`，例 `prompt:rewrite_q` |

**核心字段**：每条配置都同时存

- `default_value`：配置文件里的原值
- `override_value`：用户通过 REST 改过后的值

**业务代码统一只读 `override_value`**，不感知 `default_value`。

## 四、DB 表（走 RagMgr）

DB 走现有 `RagMgr`，表名沿用项目 `tbl_` 命名约定。三张表：

### 4.1 `tbl_dm_context_config_version`（全局版本号）

| 字段 | 类型 | 说明 |
|---|---|---|
| config_type | VARCHAR(32) | 主键，`common` 或 `prompt` |
| version | VARCHAR(32) | 当前文件 version |
| updated_at | DATETIME | 最后更新时间 |

### 4.2 `tbl_dm_context_common_config`（通用配置项）

| 字段 | 类型 | 说明 |
|---|---|---|
| id | VARCHAR(256) | 主键，格式 `{type}:{name}`，例 `com_params:aging_period` / `query_config:mode` / `com_params:a.b` |
| type | VARCHAR(64) | 配置段，对应 `ConfigMgr::GetAllConfigTypes()` 返回的 type |
| name | VARCHAR(192) | 配置项名（嵌套对象用 `.` 扁平，例 `a.b`） |
| value_kind | VARCHAR(16) | `string` 或 `array`：`array` 表示整段数组以 JSON 字符串存于 value 字段（对应现有 `UpdateArrayConfig`）|
| default_value | TEXT | 文件原值（`array` 时为 JSON 字符串） |
| override_value | TEXT | 用户改后的值（业务读这个） |
| updated_at | DATETIME | 最后更新时间 |

### 4.3 `tbl_dm_context_prompt_config`（Prompt 配置项）

| 字段 | 类型 | 说明 |
|---|---|---|
| id | VARCHAR(256) | 主键，格式 `prompt:{name}` |
| name | VARCHAR(192) | prompt 名 |
| default_value | TEXT | 文件原值 |
| override_value | TEXT | 用户改后的值 |
| updated_at | DATETIME | 最后更新时间 |

> Schema 文件参考现有 `database/qa_short_memory_schema.json` 风格新增 `database/dm_context_config_schema.json`。

## 五、配置文件格式

### 5.1 `dmcontextservice_config.json`（在现有结构基础上加 `version`）

```json
{
    "version": "1.0.0",
    "com_params": {
        "aging_period": "30"
    },
    "aging_policy": [
        { "aging_policy_id": "1", "memory_type": "user_conversion", "...": "..." }
    ],
    "memory_tbl_map": {
        "qa_short_memory": "tbl_qa_short_memory"
    },
    "rerank_config": {
        "url": "http://192.168.17.47:3025",
        "top_k": "5"
    },
    "query_config": {
        "mode": "quality",
        "max_db_query_turns": "20"
    },
    "forget_memory_commands": [ "忘掉我的记忆。", "..." ]
}
```

### 5.2 `prompt.json`（同样加 `version`）

```json
{
    "version": "1.0.0",
    "data": [
        { "name": "rewrite_q", "prompt": "..." },
        { "name": "summarize", "prompt": "..." }
    ]
}
```

### 5.3 扁平化规则

- **嵌套对象**：以 `.` 为分隔符递归扁平。
  - `com_params.aging_period` → `id = "com_params:aging_period"`，`type = "com_params"`，`name = "aging_period"`，`value_kind = "string"`
  - 深嵌套 `com_params.a.b` → `id = "com_params:a.b"`，`type = "com_params"`，`name = "a.b"`，`value_kind = "string"`
- **数组类**（如 `aging_policy` / `forget_memory_commands` / `database`）：整段数组作为 JSON 字符串存。约定：
  - `id = "{type}:_array"`（用 `_array` 占位避免与子字段冲突）
  - `type = "{type}"`，`name = "_array"`，`value_kind = "array"`
  - `default_value` / `override_value` 都是 JSON 数组字符串
- **顶层 `version`**：不进 common 表，进 `tbl_dm_context_config_version`。

## 六、运行时内存结构

```
ConfigStore（C++ 单例）
├── common
│     version: "1.0.0"
│     entries: Map<(type, name), ConfigEntry>
│       例 ("com_params", "aging_period")  → { default:"30",  override:"30",  kind:"string" } 未修改
│       例 ("rerank_config", "top_k")     → { default:"5",   override:"10",  kind:"string" } 已修改
│       例 ("forget_memory_commands","_array") → { default:"[\"...\",...]", override:"[...]", kind:"array" }
│
└── prompt
      version: "1.0.0"
      entries: Map<name, ConfigEntry>
        例 "rewrite_q" → { default:"你是xxx...", override:"你是xxx..." }
```

```cpp
struct ConfigEntry {
    std::string defaultValue;
    std::string overrideValue;
    std::string valueKind;  // common 用："string" / "array"；prompt 不用
};

class ConfigStore {
public:
    static ConfigStore& GetInstance();

    // 读：业务代码统一通过这里
    std::string GetCommon(const std::string& type, const std::string& name) const;
    std::string GetCommonArray(const std::string& type) const;  // 数组段：name 固定 "_array"
    std::string GetPrompt(const std::string& name) const;

    // 写：REST 调用
    bool SetCommon(const std::string& type, const std::string& name, const std::string& value);
    bool SetCommonArray(const std::string& type, const std::string& jsonArray);
    bool SetPrompt(const std::string& name, const std::string& value);

    // 启动调用一次，内部完成 reload + 表初始化
    bool Init();

private:
    mutable std::shared_mutex m_mu;
    std::map<std::pair<std::string, std::string>, ConfigEntry> m_common;
    std::map<std::string, ConfigEntry> m_prompt;
    std::string m_commonVersion;
    std::string m_promptVersion;

    // 异步写 DB
    void EnqueueDbWrite(/* DbWriteTask */);
    std::thread m_writerThread;
    /* queue + cv + stop flag */
};
```

### 8.2 业务读（O(1)）

`ConfigMgr::GetCommonParamsByKey("aging_period")` 内部：

```cpp
std::string ConfigMgr::GetCommonParamsByKey(const std::string& key) {
    return ConfigStore::GetInstance().GetCommon("com_params", key);
}
```

数组类同理：

```cpp
std::vector<std::string> ConfigMgr::GetForgetMemoryCommands() {
    auto json = ConfigStore::GetInstance().GetCommonArray("forget_memory_commands");
    // 解析 json 数组返回 vector<string>

```

### 8.3 REST 写

`ConfigService::UpdateConfig` 内部：

```cpp
1. 解析参数 (type, name, value)
2. ConfigStore::GetInstance().SetCommon(type, name, value)   // 同步更新内存
   └─ 内部 EnqueueDbWrite，立即返回
3. SendResponse(200, "updated successfully")
```

异步写线程：

```
loop:
  从 queue 取一个 DbWriteTask
  调 ConfigDbStore::UpdateOverride(id, override_value)
  失败：LOG_ERR，按用户决定不重试（接受失败窗口）
```

### 8.4 失败窗口说明（已确认接受）

- 服务在内存更新后、DB 写入前崩溃，重启后该次修改丢失。
- 重启 reload 时该项的 DB `override_value` 仍是旧值，被判定为"未修改"或"已修改但旧值"，业务读到旧值。
- 接受这一窗口；调用方需要强一致时通过"PUT 后 GET"自校验。

## 九、Reload 算法（核心）

仅当**文件 `version` 与 DB `config_type` 表中存的 version 不一致**（含 DB 为空的首次启动）时执行。

```
1. 解析新文件，得到一组 (id, name, kind, new_value) 项
2. 对每个新项：
     在 DB 中查 id：
       存在：
         如果 override_value == 旧 default_value（用户未改）：
           default = new_value
           override = new_value         （两者同步）
         否则（用户改过）：
           default = new_value
           override 保持不变             （保留用户修改）
       不存在：
         插入：default = new_value，override = new_value
3. （可选）DB 中存在但新文件里没有的项 → 标记或删除（见 §九.2）
4. 更新 tbl_dm_context_config_version 的 version 为新值
5. 装入 ConfigStore 内存
```

### 9.1 示例

**初始 DB**

```
config_version:  common = "1.0.0"
common_config:
  com_params:aging_period       default="30",  override="30"
  rerank_config:top_k           default="5",   override="5"
```

**用户 PUT `rerank_config:top_k = "10"`**

```
common_config:
  com_params:aging_period       default="30",  override="30"
  rerank_config:top_k           default="5",   override="10"   ← 用户改过
```

**运维替换文件，version 升至 1.1.0，aging_period 改成 60，新增 query_config:mode**

```json
{ "version": "1.1.0",
  "com_params": { "aging_period": "60" },
  "rerank_config": { "top_k": "5" },
  "query_config": { "mode": "quality" } }
```

**Reload 执行**

| id | override vs 旧 default | 操作 |
|---|---|---|
| com_params:aging_period | "30" == "30" → 未修改 | default="60"，override="60"（同步） |
| rerank_config:top_k | "10" != "5" → 已修改 | default="5"（不变），override="10"（保留） |
| query_config:mode | 不存在 | 插入，default="quality"，override="quality" |

**Reload 后 DB**

```
config_version:  common = "1.1.0"
common_config:
  com_params:aging_period       default="60",  override="60"
  rerank_config:top_k           default="5",   override="10"   ← 保留
  query_config:mode             default="quality", override="quality"
```

业务读：`com_params:aging_period → "60"`，`rerank_config:top_k → "10"`。

### 9.2 文件中删除的项

约定：reload 时，**DB 中存在但新文件中没有的 id 先标记不删除**（避免误删用户尚在使用的配置），由后续运维显式触发清理接口（不在本方案）。

## 十、REST API（路径与现状一致，行为微调）

完全沿用 `route_mgr.cpp` 现有注册：

| 方法 | 路径 | 行为变化 |
|---|---|---|
| GET | `/api/v1/contexts/configs` | 不变，返回 type 列表 |
| GET | `/api/v1/contexts/configs/items?type=xxx` | 不变，返回该 type 的所有项（来自内存） |
| GET | `/api/v1/contexts/configs/item?type=xxx&name=yyy` | 不变，返回 `override_value` |
| PUT | `/api/v1/contexts/configs/item?type=xxx&name=yyy` | 内部调 `ConfigStore.SetCommon`，不再直接写文件 |
| GET | `/api/v1/contexts/prompts/item?name=xxx` | 不变，返回内存值 |
| PUT | `/api/v1/contexts/prompts/item?name=xxx` | 内部调 `ConfigStore.SetPrompt` |
| PUT | `/api/v1/contexts/prompts` | 批量调 `SetPrompt`，每条独立异步落 DB |

返回格式保持现状。

> 当前 `ConfigService::UpdateConfig` 在外部传 `value` 是 JSON 数组时走 `UpdateArrayConfig` 分支，本方案在 `ConfigStore.SetCommonArray` 中保持同样语义：把整段 JSON 写到 `value_kind = "array"` 的 entry。

## 十一、main.cpp 启动顺序调整

现有顺序：

```
1. RouteMgr::InitRestRouter
2. FrameworkStarter::InitFramework
3. ConfigMgr::InitConfigMgr               ← 此时 RagMgr 未初始化
4. RagMgr::Init                            ← retry 10×5s
5. DatabaseMgr::InitDatabaseMgr
6. MicroserviceMgr::Init
7. ModelMgr::Init                          ← retry 10×5s
```

调整后：

```
1. RouteMgr::InitRestRouter
2. FrameworkStarter::InitFramework
3. RagMgr::Init                            ← 先做
4. ConfigMgr::InitConfigMgr                ← 内部已切到 ConfigStore，依赖 RagMgr
   └─ ConfigStore::GetInstance().Init()    ← 在 ConfigMgr::InitConfigMgr 内部首步
5. DatabaseMgr::InitDatabaseMgr
6. MicroserviceMgr::Init
7. ModelMgr::Init
```

> `RagMgr::Init` 失败已有 10×5s retry；`ConfigStore::Init` 失败属于致命错误，按现有 `CHECK_ACT_RET` 模式 `return -1` 退出。

- `RewriteRuleService::call_once` 的解阻塞——这是热生效问题，与持久化机制独立。本方案完成后，业务方在迁移参数时统一改成"读 ConfigStore"，自然规避 `call_once`。
- `ConfigMgr::GetConfig` 重复加锁的修复——本方案重写读路径时顺手处理。
- `SaveConfigToFile` 不支持新增 key——本方案直接弃用文件回写，问题自动消失。

## 十三、风险与注意事项

1. **RagMgr API 在本目录不可见**：表创建、行 CRUD 的具体调用需要外层 CoreMind 平台树确认。建议先在外层小写一个 spike 验证。
2. **数组类 reload 比较**：`override == 旧 default` 的字符串等价比较对 `array` 类型敏感于字段顺序。`ConfigLoader` 比较时对 `value_kind == "array"` 的项做 JSON 等价比较（解析后比较 DOM），不能直接 `==` 字符串。
3. **失败窗口**（已确认接受）：内存写入与 DB 写入之间崩溃会丢失修改；重启后表现为"用户修改未生效"，调用方可重试。
4. **配置文件 `version` 字段必须由开发/运维主动维护**：本方案上线前需要在 `dmcontextservice_config.json` 和 `prompt.json` 增加 `version` 字段；运维流程文档需更新。
5. **prompt.json 现有结构未确认**：本目录看不到内容（该文件 25KB，可能是顶层 key-value 也可能已是 `data` 数组）。落地前需对照现有解析逻辑确定是否需要结构调整或加兼容层。
6. **ConfigStore 单例 Init 的并发**：业务请求线程必须在 `ConfigStore::Init()` 完成后才能 Get；现有 main.cpp 同步初始化模型保证了这一点，请勿在 Init 之前注册路由（路由在 step 1 注册，但 Framework 在 step 2 才启动监听，安全）。
7. **写 DB 失败的可观测性**：异步写失败只 `LOG_ERR`，无对外信号。建议增加单调递增的 `failed_writes_total` 计数（暴露到健康检查），便于运维巡检。
8. **Reload 时新文件解析失败**：必须保证 ConfigStore 不被半装入污染。`ConfigLoader` 应"先全量解析到临时结构，再原子替换内存"，失败则回滚到当前内存状态并返回错误。

## 十四、验收标准

| 场景 | 验收点 |
|---|---|
| 首次启动（DB 空） | 三张表自动创建；文件全量装入，default == override；config_version 写入文件 version |
| 启动 version 一致 | 不走 reload，从 DB 直接装内存；DB 中已有的 override 值生效 |
| 启动 version 升级 | 用户改过的项 override 保留，未改过的同步新值，新增项以新值插入 |
| PUT 单项 | 内存立即更新；后续 GET 返回新值；DB 在异步线程内变更 |
| PUT 数组 | `value_kind = array` 的整段 JSON 替换；GET 返回新数组 |
| PUT prompt | 内存立即更新；ModelMgr 后续调用读到新值（依赖 ModelMgr 不缓存或调 reload） |
| 服务在 PUT 后立刻崩溃 | 重启后该次修改可能丢失（已知失败窗口） |
| 配置文件无 `version` 字段 | 启动 ERROR 退出，提示运维补全（或按 "0.0.0" 兜底，二选一，待确认） |
